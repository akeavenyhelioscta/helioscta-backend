# Documentation Skill

How to maintain and extend the Docusaurus documentation site in `docs/`.

## Stack

- **Framework:** Docusaurus 3 (React-based, Node.js)
- **Config:** `docusaurus.config.js` (project root)
- **Sidebar:** `sidebars.js` (project root)
- **Content:** `docs/` directory (Markdown)
- **Custom CSS:** `src/css/custom.css`
- **CI/CD:** `.github/workflows/docs.yml` deploys to GitHub Pages on push to `main`

## Local Development

```bash
npm install     # first time only
npm start       # live preview at localhost:3000
npm run build   # production build to build/
npm run serve   # serve production build locally
```

## Theme

- Dark mode is the default (`colorMode.defaultMode: "dark"`)
- True black background (`#000`) via CSS custom properties in `src/css/custom.css`
- Full-width content layout (no max-width container)
- Active sidebar items highlighted in amber (`#ffc245`)
- High-contrast text (`rgba(255,255,255,0.92)` on `#000`)

## Sidebar Navigation Rules

- Labels should be concise, 2-4 words (e.g., "DA LMPs" not "PJM Day-Ahead Hourly LMPs")
- Defined in `sidebars.js` using explicit doc IDs (not autogenerated)
- Group related pages under categories; keep hierarchy max 3 levels deep
- Reference pages (Glossary, Owners & SLAs) in a collapsed=false "Reference" category
- Use "Scrapes" and "dbt Views" as standard subsection names

## Content Conventions

- Every domain overview opens with a 1-2 sentence summary
- Follow with `**Use this page** to...` to orient the reader
- Add "Next steps" blocks at page bottoms linking to related resources
- Admonitions use `:::note` / `:::tip` / `:::warning` syntax (NOT `!!!`)
- The home page (`docs/README.md`) has `slug: /` front matter to serve as the root
- Terminology must match `docs/glossary.md`

## Page Templates

### Domain Overview

```markdown
# Domain Name

One-two sentence summary of what data this domain covers and why it matters.

**Use this page** to find scrape scripts, raw tables, and dbt views for [domain].

## Data Source
## Scrape Inventory
## dbt Views
## Known Caveats
## Owner
```

### Scrape Card

```markdown
## Scrape Card

| Field | Value |
|-------|-------|
| **Script** | path |
| **Source** | API name |
| **Target Table** | schema.table |
| **Trigger** | Scheduled / Event-Driven |
| **Cadence** | frequency |
| **Freshness** | lag description |

## Business Purpose
## Data Captured
## Primary Key
## Downstream
## Known Caveats
```

## Adding a New Domain

1. Create `docs/domains/<domain>/overview.md`
2. Create `docs/domains/<domain>/scrapes/` and `docs/domains/<domain>/dbt-views/` as needed
3. Add the domain as a category in `sidebars.js` with doc IDs
4. Add entries to `docs/dbt-cleaned-catalog.md`
5. Update `docs/executive-summary.md` tables

## Adding a New Scrape Page

1. Create the scrape card in `docs/domains/<domain>/scrapes/<scrape-name>.md`
2. Add its doc ID to the domain's Scrapes category in `sidebars.js`
3. Link from the domain overview's scrape inventory table
4. Link from the Data Catalog if it feeds a dbt view

## Deployment

- Pushes to `main` that touch `docs/`, `src/css/`, `docusaurus.config.js`, `sidebars.js`, or `package.json` trigger the GitHub Actions workflow
- Workflow: install deps, `npm run build`, deploy to GitHub Pages
- Site URL: `https://akeavenyhelioscta.github.io/helioscta-backend/`
- Requires a `github-pages` environment in repo Settings > Environments

## QA Checklist

- Dark mode: true black background, high-contrast text, amber active nav
- Light mode: toggle works, no regressions
- Mobile: sidebar collapses, tables scroll horizontally
- Links: `onBrokenLinks: "throw"` catches broken links at build time
- Search: Docusaurus built-in search works out of the box
- Admonitions: use `:::` syntax, not `!!!`
